Setting up YouTube on Playbill

A one-time, 10-minute setup that lets YouTube work on your Playbill using your own Google account.

Why this exists. Google reserves "TV-style YouTube clients" for OEM partners (Roku, Apple, Samsung, etc.). Smaller third-party clients like Playbill get rejected at verification, so we use the same workaround Kodi's YouTube add-on uses — each user runs their own one-person Google Cloud project. There's no shared key for Google to revoke, no quota to share, no verification queue, and your data never leaves your rig.

What you'll do

  1. Create a Google Cloud project
  2. Enable the YouTube Data API
  3. Configure the OAuth consent screen
  4. Add yourself as a Test User
  5. Create the OAuth 2.0 Client ID
  6. Create an API Key
  7. Paste the three values into Playbill

What you'll need

  • A Google account (Gmail is fine).
  • A web browser on your phone or laptop. Easier on a regular screen than on the Playbill itself.
  • About 10 minutes.

You won't be charged anything. Everything you set up uses Google Cloud's free tier.

1Create a Google Cloud project

  1. Go to console.cloud.google.com.
  2. Sign in with your Google account.
  3. At the top of the page, click the project picker (next to "Google Cloud"). A modal opens.
  4. Click NEW PROJECT in the top-right of the modal.
  5. Set Project name to anything you want — e.g. My Playbill.
  6. Click CREATE.
  7. Wait ~30 seconds. The project picker refreshes with your new project selected.

2Enable the YouTube Data API v3

  1. Left nav → APIs & Services → Library. (If you don't see the side nav, click the ☰ icon top-left.)
  2. Search for YouTube Data API v3.
  3. Click the result.
  4. Click ENABLE. Wait ~10 seconds.

3Configure the OAuth consent screen

  1. Left nav → APIs & Services → OAuth consent screen.
  2. User Type: select External, then click CREATE.
  3. Fill in the required fields:
    • App name: anything (e.g. My Playbill).
    • User support email: your Gmail.
    • Developer contact information: your Gmail.
  4. Click SAVE AND CONTINUE through the remaining screens until you're back at the overview. You don't need to add scopes or change anything else.

4Add yourself as a Test User

Still on the OAuth consent screen page:

  1. Find the Test users section. Click + ADD USERS.
  2. Type the Gmail address you'll be signing into YouTube with. (Can be the same as the developer email, or other family Gmail accounts — up to 100 total.)
  3. Click SAVE.
If you skip this, you'll see a "Google hasn't verified this app" warning when signing in. You can click Advanced → Go to (unsafe) to proceed, but it's nicer to just add yourself here.

5Create the OAuth 2.0 Client ID

  1. Left nav → APIs & Services → Credentials.
  2. Click + CREATE CREDENTIALS, then choose OAuth client ID.
  3. Application type: select TVs and Limited Input devices.
  4. Name: anything (e.g. Playbill TV client). Click CREATE.
  5. A modal pops up with your Client ID and Client secret. Leave that modal open — you'll paste both into Playbill in step 7.

6Create an API Key

  1. Open a new browser tab to the same Credentials page: console.cloud.google.com/apis/credentials.
  2. Click + CREATE CREDENTIALS → API key.
  3. A modal pops up with your API key. Copy it.
  4. Click EDIT API KEY in that modal (or close it and click the pencil icon next to the new key in the list).
  5. Under API restrictions:
    • Select Restrict key.
    • In the dropdown, choose YouTube Data API v3.
  6. Click SAVE.

7Paste the three values into Playbill

You should now have:

  • A Client ID (long string ending in .apps.googleusercontent.com)
  • A Client secret (~24 characters)
  • An API key (~39 characters starting with AIza)

From the Headwaters PWA on your phone:

  1. Open the PWA, tap the Playbill nav icon.
  2. Tap the YouTube tab.
  3. Paste all three values into the form.
  4. Tap Save.
  5. Tap Sign in. A 6-digit code appears.
  6. On your phone (or any other device), open youtube.com/activate and enter the code, then sign in with the Gmail account you added as a Test User in step 4.
  7. After Google says "device connected," Playbill flips to Signed in as <your channel name>.

You're done. YouTube under Sources on Playbill now works.

What to expect afterward

Quota

Google gives each Cloud project 10,000 "units" per day of YouTube Data API quota. Some reference costs:

OperationCost
Search100 units
Browse a channel page~5 units
Load a playlist~5 units
Start playback0 units (stream resolution uses yt-dlp)

A typical viewing day is 100–500 units. You'd need hundreds of searches a day to run out. If you ever do hit the limit, new searches stop working until quota resets at midnight Pacific time; already-loaded content and playback continue.

You can request a quota increase (free, ~1 business day) at console.cloud.google.com/iam-admin/quotas — filter for "YouTube Data API."

Re-consent every 7 days

While your OAuth project is in Testing mode (which is where it stays — you don't need to publish it), Google requires re-consent every 7 days. In practice that means about once a week, Playbill's YouTube tab shows "Session expired — please sign in again." Tap Sign in and walk through the 6-digit code flow.

There's no way to remove this — the only alternative would be Google verification, which they don't grant to TV-style YouTube clients. Every third-party YouTube client lives with this.

Multiple Gmail accounts

You can add up to 100 test users in step 4. If your family has separate accounts and you want each to be able to sign in (subscriptions, watch history), add all their emails as test users on the same OAuth project, then sign in/out from the Playbill YouTube tab as needed.

Multiple Playbills

If you have more than one Playbill on the rig, each one needs its own credential entry — paste the same three values into each. Quota is shared because they share one Google Cloud project.

Troubleshooting

"This app isn't verified" / "Google hasn't verified this app"

You skipped step 4 (adding yourself as a Test User), or you're signing in with a different Gmail than the one you added. Go back to step 4 and add the email you're trying to sign in with.

"Sign in failed: invalid_grant" / "Token expired"

The 7-day re-consent window elapsed. Tap Sign in again and complete the device-code flow.

"Quota exceeded"

You burned through 10,000 units in one day (rare). Wait until midnight Pacific time or request a quota increase per the link above.

"Invalid client" / "Client ID not found"

Double-check the Client ID was pasted in full and ends in .apps.googleusercontent.com. The Cloud Console truncates the display in some views; click the row in the Credentials list to see the full value.

"API key invalid"

Check that the API key starts with AIza. Confirm under API restrictions you selected YouTube Data API v3 (not "Don't restrict key"). Unrestricted keys can be rejected by Google.

The YouTube tab in Playbill says "Configure your Google Cloud credentials" but I already pasted them

Each Playbill stores its own credentials. If you set them up on Playbill A and a different Playbill B shows the same message, paste the values into Playbill B too.

What's stored where

WhereWhat
Google Cloud Console (your account)OAuth client, API key, your project quota
Playbill, file mode 0600~/.config/trailcurrent-playbill/sources/youtube/client.json — clientId, clientSecret, apiKey
Playbill, file mode 0600~/.config/trailcurrent-playbill/sources/youtube/tokens.json — OAuth refresh + access tokens
Headwaters (cloud)Nothing. None of this leaves your rig.

Playbill talks directly to Google with your credentials; nothing is proxied through Headwaters or any other service.

Why we have to do it this way

The short version: Google's API services policy explicitly bans "duplicating YouTube's experience" via the public YouTube Data API. The verification process for an app like Playbill — a TV-style YouTube client that isn't an officially-partnered TV platform — gets rejected. This isn't unique to us: Kodi's YouTube addon, Plex's third-party YouTube channels, FreeTube, ViewTube, and every other open-source YouTube client lives under the same constraint.

The "bring your own Google Cloud project" approach sidesteps the issue entirely. From Google's perspective, each Playbill owner is a developer building a one-user app for themselves. There's no shared client to verify, no quota to share, no central app for policy enforcement to flag.

It's clunky, but it's the only path that doesn't rely on either:

  • A Smart-TV-platform partnership we'd need to be Sony-sized to negotiate, or
  • An ad-stripping reimplementation that violates YouTube's terms of service and breaks every few months.

Most users finish this setup in 10 minutes and never think about it again.
